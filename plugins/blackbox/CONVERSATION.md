# 黑匣子 Skill 对话记录

## 对话主题

为代码添加黑匣子（调试和监控）能力，设计一套完整的可观测性方案。

## 第二部分：公式化表达

### 问题提出

> "我们能整理成公式语句吗。"

### 公式化修正

**原话的问题：**
> "如果没有「内存/资源监控」就无法检测到「长时间运行后浏览器崩溃」"

**问题：**
- 崩溃发生后进程已结束，无法"检测到"已发生的崩溃
- 内存监控的实际作用是**检测异常趋势**，**预防**崩溃发生

**正确的公式化表达：**

| 改进项 | 监测对象 | 捕获信号 | 驱动动作 | 达成效果 |
|-------|---------|---------|---------|---------|
| 性能监控 | 关键操作 | 耗时分布 | 定位瓶颈 | 优化性能 |
| 网络详细拦截 | HTTP 请求 | 完整数据 | 快速定位 | 修复 API |
| 自动重试 | 操作失败 | 失败信号 | 延迟重试 | 提高稳定性 |
| 断言工具 | 页面状态 | 验证条件 | 立即报错 | 快速失败 |
| SSE 调试 | 流式数据 | 数据片段 | 追踪断点 | 修复处理 |
| 内存监控 | 内存使用 | **增长趋势** | **主动重启** | **预防崩溃** |

**关键区别：**
- ❌ 检测到崩溃（太晚了）
- ✅ 检测到增长趋势，预防崩溃（正确）

## 第五部分：黑匣子的设计理念

### 上层理念的讨论

**核心哲学：**
```
代码一直在运行，但我们不知道它在经历什么。
黑匣子让代码的"经历"可以被看见。
```

**四大上层理念：**

#### 1. 存在性 vs 可见性
- **理念：** 让不可见变为可见，让隐性变为显性

#### 2. 事后追溯 vs 实时监控
```
传统监控：实时查看，产生噪音
黑匣子：   默默记录，事后追溯
```
- **理念：** 平时沉默，出事说话

#### 3. 非侵入性
```
观察者效应：观察行为会改变被观察对象
黑匣子目标：记录而不干扰
```
- **理念：** 旁观而不干预

#### 4. 完整性 vs 选择性
```
问题：我们不知道将来需要什么数据
答案：所以记录一切
```
- **理念：** 宁多勿少，按需查询

---

### 开发 vs 生产环境

**重要认识：**

> "理论上我们只在开发时需要他。不是生产环境，所以生产时，不处罚，是在开发阶段，选择性触发。开发环境默认是模拟生产环境的。"

| 环境 | 黑匣子状态 | 说明 |
|-----|-----------|------|
| **生产** | 完全关闭 | 不影响性能和稳定性 |
| **开发** | 按需开启 | 默认关闭，调试时开启 |
| **测试** | 默认开启 | 帮助发现测试问题 |

## 第七部分：注释规范设计

### 重要设计决策

> "埋下去，并不是给人看的。平时不需要查看，只有出事故的时候，才会去观察它。"

### 注释标记格式

```typescript
// 📦 [BB] <类型> <说明>
```

### 类型分类

| 标记 | 类型 | 说明 |
|-----|------|------|
| `📦 [BB] 性能` | 性能埋点 | 记录操作耗时 |
| `📦 [BB] 网络` | 网络埋点 | 记录请求/响应 |
| `📦 [BB] 资源` | 资源埋点 | 记录内存/CPU |
| `📦 [BB] 错误` | 错误埋点 | 记录异常信息 |
| `📦 [BB] 状态` | 状态埋点 | 记录页面状态 |
| `📦 [BB] 快照` | 快照埋点 | 保存现场数据 |

### 搜索命令

```bash
# 搜索所有黑匣子埋点
grep -r "📦 \[黑匣子\]" src/

# 搜索特定类型
grep -r "📦 \[黑匣子\] 性能" src/

# 排除黑匣子查看正常代码
grep -v "📦 \[黑匣子\]" src/file.ts
```

## 第八部分：知识层级结构

### 问题提出

> "SKILL 之上，是哲学么。还是有其他，以更好的思想补充"

### 知识层级设计

```
┌─────────────────────────────────────────────────────────────┐
│                     Philosophy（哲学/理念）                    │
│                      为什么存在                              │
├─────────────────────────────────────────────────────────────┤
│                     Principles（原则）                        │
│                      核心指导原则                              │
├─────────────────────────────────────────────────────────────┤
│                      Patterns（模式）                         │
│                     可复用的解决方案                            │
├─────────────────────────────────────────────────────────────┤
│                       Skills（技能）                          │
│                      具体操作指南                              │
└─────────────────────────────────────────────────────────────┘
```

### 四大原则

| 原则 | 定义 | 约束 |
|-----|------|------|
| **沉默原则** | 平时不输出，不产生噪音 | 默认关闭，按需开启 |
| **完整原则** | 不知道需要什么，所以记录一切 | 结构化存储，查询时过滤 |
| **隔离原则** | 黑匣子代码与业务代码隔离 | 通过注释标记，环境变量控制 |
| **透明原则** | 埋点代码清晰可识别 | 统一注释格式 |

### 原则优先级

```
1. 沉默原则（最高优先级）- 不干扰运行
   ↓
2. 隔离原则（其次）- 代码可维护
   ↓
3. 透明原则 - 代码可识别
   ↓
4. 完整原则（最低优先级）- 数据充分性
```

## 关键设计决策总结

### 命名决策

- 最终使用 "blackbox" 作为 skill 名称
- 强调"飞行记录仪"的比喻

### 环境策略

- 生产环境：默认关闭
- 开发环境：按需开启，默认关闭
- 测试环境：默认开启

### 代码规范

- 使用 `// 📦 [BB] <类型>` 注释标记
- 通过环境变量 `BLACKBOX_ENABLED` 控制
- 默认不输出到控制台

### 知识组织

- 四级层级：Philosophy → Principles → Patterns → Skills
- 分离设计理念与具体操作
- 渐进式披露，按需加载
