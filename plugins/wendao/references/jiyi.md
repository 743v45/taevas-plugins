# 记忆系统使用规则

## 概述

问道体系可以结合用户的记忆系统，主动发现用户近期关注的概念或技术，提供更个性化的追问体验。

→ [返回 SKILL.md](../SKILL.md)

---

## 何时使用记忆

**触发条件**：用户没有明确描述要分析什么事物时

在"交互式使用流程"的第一步，如果用户未指定分析对象，先检查记忆中是否有相关主题。

---

## 记忆筛选原则

从记忆中选择主题时，遵循两个维度：

### 1. 新鲜程度

- **优先级**：越新的记忆越优先
- **判断依据**：记忆文件的修改时间、记录的时间戳
- **原因**：用户最近关注的事物更可能是当前想深入探索的方向

### 2. 可讨论程度

- **优先级**：适合用问道体系深度追问的主题优先
- **判断依据**：
  - 技术概念：缓存、API、状态机、事件驱动等
  - 抽象模式：抽象、多态、闭包、异步等
  - 系统设计：用户登录、订单系统、权限管理等
  - 通用概念：黑匣子、区块链、DNA、社交网络等
- **排除**：
  - 纯粹的配置信息
  - 临时性的操作指令
  - 缺乏抽象空间的具体细节

---

## 主题数量处理

根据筛选结果灵活处理：

| 筛选结果 | 处理方式 |
|---------|---------|
| 找到 0 个相关主题 | 使用通用举例，让用户自行选择 |
| 找到 1 个相关主题 | 二选一：该主题 vs 其他 |
| 找到多个相关主题 | 多选 + 其他选项 |

---

## 发散式举例引导

当用户选择某个记忆中的主题后，提供发散式举例，展示问道体系如何展开分析。

**以"缓存"为例：**

```
- 缓存与 CPU 缓存、浏览器缓存、DNS 缓存是否同构？
- 缓存能否迁移到人际关系领域？（人际关系缓存）
- 缓存的核心结构是什么？（慢速存储 + 快速存储 + 替换策略）
- 什么情况不适合用缓存？（频繁变更的数据）
```

**目的**：让用户快速理解问道体系的追问方式，激发思考方向。

---

## 与记忆类型的对应

不同类型的记忆可能包含不同的主题来源：

| 记忆类型 | 可能的主题来源 | 示例 |
|---------|--------------|------|
| user | 用户学习过的技术、感兴趣的概念 | "用户是数据科学家" → 可能关注数据处理、可视化 |
| project | 项目中涉及的技术栈、架构模式 | "使用 Tauri 2.x 构建" → Tauri 相关概念 |
| feedback | 用户反复强调的偏好或原则 | "避免过度抽象" → 抽象的边界问题 |

---

## 注意事项

1. **不强制依赖记忆**：没有记忆时也能正常工作，使用通用举例
2. **不过度推断**：只选择明显相关、适合追问的主题
3. **保持开放**：始终提供"其他"选项，允许用户自由选择
4. **自然呈现**：将记忆主题作为建议，而非限制

---

## 示例流程

```
用户调用 /wendao（无参数）

→ 检查记忆，发现用户最近研究过"缓存"、"状态机"

→ AskUserQuestion：
  "注意到您最近在探索 缓存、状态机。您想用问道体系深入了解哪一个？"
  选项：缓存、状态机、其他

→ 用户选择"缓存"

→ 展开发散式举例，进入第二步（确定深入层级）
```
